= Toolchain Reference

This file documents the exact toolchain versions used to build and
test the bypass firmware.  It serves the same purpose as a Python
requirements.txt: a human-readable record of what the project was
validated against, so a new developer (or CI system) can reproduce
the exact environment.

Classic AVR firmware targets rebuild whenever they are requested, ensuring the
current compiler, flags, and source tree are always consumed.  If you update a
tool, re-run `make clean && make test` before relying on its output.

== Development platform

  OS:           Linux 6.12.33 (TrueNAS production kernel), x86_64
  Distribution: Ubuntu 24.04 (Noble) base packages

== AVR toolchain (cross-compile to MCU target)

  avr-gcc:      7.3.0  (package: gcc-avr)
  avr-libc:     2.0.0+Atmel3.7.0-1  (package: avr-libc)
  avrdude:      not installed on build host; flash via dedicated programming
                host/rig.  Recommended: avrdude 6.3 or 7.x with USBasp or
                similar ISP programmer.

  Minimum avr-gcc version: 4.9 (for _Static_assert / C11 support with -std=gnu11)
  Recommended: 7.x or later (best -fshort-enums / LTO support for tiny parts)

== PIC toolchain (PIC10F322, optional second target)

The PIC10F322 port (`make pic`, `make pic-test`,
`make pic-test-target-variants`) is a SEPARATE, optional toolchain, entirely
independent of the AVR build above and deliberately NOT part of `make test` (the
AVR pre-hardware gate).  Individual PIC targets skip cleanly when their tools
are absent, so the AVR workflow is unaffected if these are not installed.  CI
and release use `STRICT_TOOLS=1` and the fail-closed `pic-test-target-variants`
aggregate so missing PIC tooling cannot produce a false green release gate.

  MPLAB XC8:    V3.10  (Microchip MPLAB XC8 C Compiler; build date Aug 13 2025)
                install dir: /opt/microchip/xc8/v3.10
                driver:      /opt/microchip/xc8/v3.10/bin/xc8-cc
                override the path with PIC_CC=... on the make command line.

  Device pack:  PIC10-12Fxxx_DFP  v1.9.189  (Microchip device family pack)
                Supplies the PIC10F322 device-support + register headers
                (proc/pic10f322.h) that the XC8 base install does NOT include.
                install dir: /opt/microchip/mdfp/PIC10-12Fxxx_DFP/1.9.189
                NOTE: -mdfp must point at the pack's xc8/ SUBDIR, not the pack
                root (pointing at the root gives XC8 error 2104):
                  -mdfp=/opt/microchip/mdfp/PIC10-12Fxxx_DFP/1.9.189/xc8
                override with PIC_DFP=... on the make command line.

  gpsim:        0.32.1  (Ubuntu package: gpsim)
                Instruction-level PIC simulator used by `make pic-test-gpsim`
                for register-level functional tests.  The libgpsim harnesses
                below add the PIC lock-step, fault-injection, and target-I/O
                gates.
                install: apt install gpsim
                note: gpsim 0.32.1 DOES support the p10f322; its I/O pins are
                named ra0..ra3 (lower-case) in stimulus/attach scripts.

  libgpsim:     gpsim-dev + libglib2.0-dev + g++ + pkg-config  (Ubuntu packages)
                Needed by `make pic-test-fault` (SFR / pull-up / ctx_
                fault-injection), `make pic-test-lockstep` (built-HEX live SRAM
                vs model), `make pic-test-io` (GPIO transition and pulse timing),
                and `make pic-test-soak` (long-duration soak). These harnesses
                LINK libgpsim instead of scripting the gpsim CLI; the runtime
                `gpsim` package ships no headers, the C++ harnesses need g++,
                and the glib include/link flags are discovered through
                pkg-config.
                install: apt install gpsim-dev libglib2.0-dev g++ pkg-config
                note: gpsim-dev's headers `#include <glib.h>`, which is NOT
                pulled in by gpsim-dev, so libglib2.0-dev must be installed
                explicitly (parallel to the libelf-dev gotcha for simavr).
                The build passes -isystem for the gpsim headers (they emit their
                own -Woverloaded-virtual warnings).  The standalone libgpsim
                targets skip cleanly when dev packages or the XC8-built HEX are
                absent; use `make pic-test-target-variants STRICT_TOOLS=1` when
                a missing or skipped target must fail closed.

  cppcheck:     2.13.0  (the SAME binary as the AVR analysis, below)
                `make pic-analyze` runs it with --platform=pic8-enhanced over the
                real XC8 + DFP register headers.  No install beyond the XC8/DFP
                packs.

XC8 / device-pack behaviours relied on (proven against V3.10 + DFP v1.9.189):

  - XC8 V3.10 compiles C99 only (no C11); the firmware's static_assert is shimmed
    to the _Static_assert keyword in src/bypass_static_assert.h.
  - XC8 rejects -fshort-enums, and its enum sizing is TWO-FACED: the clang
    front end (which evaluates sizeof and _Static_assert) sizes enums as int,
    so sizeof(debounce_context_t) is 5 -- but the CODE GENERATOR packs each
    enum to the smallest sufficient type, so _ctx_ actually assembles to
    `ds 3` with fields at byte offsets +0/+1/+2 (verified, including with
    volatile members, i.e. it is systematic layout, not an optimization).
    Consequences: (a) a static_assert CANNOT pin the real layout -- the PIC
    libgpsim tests assert `_ctx_: ds 3` from the generated .s instead, since
    their ctx_ SRAM offsets and the RAM figures in DESIGN_DOCUMENTATION.adoc
    depend on it; (b) never take sizeof of an
    enum-bearing type in PIC-shell code (e.g. memset(&ctx_, 0, sizeof ctx_)
    would write 5 bytes over a 3-byte object).  The atomicity angle is
    harmless either way: the PIC shell is single-threaded (polled tick, no
    ISR-shared state), so the 8-bit atomicity the AVR build gets from
    -fshort-enums is not needed.
  - const-qualified locals are placed in program ROM, so a const local
    initialised from a runtime call is rejected; the PIC shell drops that const.
  - _XTAL_FREQ (for the drivers' __delay_ms) is supplied via -D in the Makefile,
    parallel to the AVR's -DF_CPU.

Device budget (PIC10F322): 512 words of flash, 64 bytes of RAM.  `make pic`
enforces the flash-word budget per variant; see "Resource Utilization" in
DESIGN_DOCUMENTATION.adoc for the measured figures.

XC8 and the device pack are Microchip downloads, NOT apt packages:

  XC8:         https://www.microchip.com/en-us/tools-resources/develop/mplab-xc-compilers
  Device pack: install via MPLAB X (Tools -> Packs), or download the
               PIC10-12Fxxx_DFP pack from Microchip's pack repository.

== Simulator (integration tests)

  simavr:       1.6+dfsg-3build2  (Ubuntu package: simavr, libsimavr-dev)
  install:      apt install simavr libsimavr-dev libelf-dev
  note:         the sim test harness links against libelf (-lelf); libelf-dev
                is NOT pulled in by libsimavr-dev, so it must be installed
                explicitly or the test/avr/test_sim_* link step fails.
  note:         ATtiny13 WDT-reset emulation is NOT available in simavr 1.6;
                the ATtiny85 path is used for WDT tests.  See test/avr/test_sim.c
                for the #ifdef TARGET_T85 stubs.

  yasimavr:     0.1.6 + 2 vendored patches  (PyPI sdist, GPL-3.0; NOT apt)
                Scriptable AVR8X/XT simulator used by the ATtiny202 (AVR-XT)
                dynamic harness -- `make attiny202-sim` (functional footswitch
                -> LED), `attiny202-fault` (critical-SFR / state fault
                injection), `attiny202-soak` (long-duration liveness).  It is
                the AVR-XT analogue of simavr for the classic parts and gpsim
                for the PIC: simavr 1.6 and QEMU do not model the AVR8X core, so
                the ATtiny202 shell would otherwise have no lock-step / soak /
                fault coverage (only static analysis + hardware).  The three
                targets are STANDALONE and skip cleanly when the venv is absent,
                exactly like the PIC gpsim targets, so the AVR `make test` is
                unaffected if yasimavr is not built.
  build:        scripts/fetch_yasimavr.sh -- the yasimavr counterpart of
                scripts/fetch_attiny_dfp.sh.  Builds a PATCHED yasimavr into a
                project-local, gitignored venv (third_party/yasimavr/venv) from
                the pinned sdist (yasimavr==0.1.6, SHA-256 verified) plus the two
                vendored bug-fix patches in third_party/yasimavr/patches/.
                Idempotent (stamped on version + sdist SHA + patch-set SHA).
  needs:        python3 + python3-venv + python3-dev and a C++ compiler (it
                compiles yasimavr's C++/SIP extension modules from source,
                ~75 s), plus curl/wget and network access to PyPI.  Portable
                across a stripped-ensurepip host (creates the venv --without-pip
                and bootstraps get-pip) and a normal one.
  install:      sudo apt install python3 python3-venv python3-dev g++ curl
                scripts/fetch_yasimavr.sh
  patches:      two upstream yasimavr bugs (reported upstream) block our
                fuse-locked-WDT firmware and are carried as the vendored patches:
                (0001) the tinyAVR 0-series device builder omits the WDT
                peripheral, so any WDT register access crashes the simulated CPU;
                (0002) ArchXT_WDT maps WINDOW=OFF to a 4-clock window instead of
                0, resetting a correctly-petted WDT ~every pet.  Bug 0002 lives
                in compiled C++, so a patched BUILD is required -- a runtime
                monkeypatch cannot reach it.
  scope:        yasimavr models the ATtiny202 TCB0 tick, the fuse-locked WDT
                (real reset + STATUS.LOCK), RSTCTRL and PORTA (input + output)
                well enough to run the REAL built image.  It treats the shell's
                interrupts-off force-reset spin (cli; for(;;){}) as a terminal
                halt, so it cannot show the watchdog PHYSICALLY completing that
                reset (a hardware guarantee); the fault test asserts the sanity
                gate ENTERING the force-reset path instead, and the watchdog
                liveness reset (out of SLEEP) is observed directly.  The pure
                debounce core keeps its full host + formal coverage regardless.
                See test/avr/sim_attiny202.py.
  builds on:    the ATtiny_DFP device files (scripts/fetch_attiny_dfp.sh, pinned
                atpack) that `make attiny202` needs to COMPILE the image; those
                are a Microchip download, fetched on demand, not committed.

== Host toolchain (native test harnesses)

  gcc:          13.3.0  (Ubuntu 13.3.0-6ubuntu2~24.04.1)
  clang:        18.1.3  (Ubuntu LLVM 18.1.3, package: clang-18)

  Either compiler works for all test/* host binaries.  The Makefile uses gcc
  by default (HOSTCC ?= gcc); set HOSTCC=clang to override.

== Static analysis

  clang-tidy:   18.1.3  (from LLVM 18 packages)
  cppcheck:     2.13.0  (Ubuntu package: cppcheck)
  clang --analyze: provided by clang 18 (no separate install)

  All three are wired into `make analyze`.  Warnings are treated as errors.

== Optional (symbolic execution)

  KLEE:         not installed; test/formal/test_symbolic.c compiles and runs under
                the host gcc as an exhaustive enumerator when USE_KLEE is not
                defined.  To use with KLEE, build KLEE from source against
                LLVM 18 and compile test_symbolic.c with:
                  clang -emit-llvm -DUSE_KLEE ... -o test_symbolic.bc
                  klee test_symbolic.bc

== Reproducing the environment (Ubuntu 24.04)

  sudo apt install gcc-avr avr-libc simavr libsimavr-dev libelf-dev \
                   gcc clang clang-tidy cppcheck

  Optional analysis tools:
  sudo apt install valgrind lcov gcovr

  Optional PIC10F322 port (`make pic` / `make pic-test` /
  `make pic-test-target-variants`):
  sudo apt install gpsim          # the PIC simulator (apt)
  # For the libgpsim fault/lock-step/target-I/O gates and the soak:
  sudo apt install gpsim-dev libglib2.0-dev g++ pkg-config
  # XC8 V3.10 + the PIC10-12Fxxx_DFP pack are separate Microchip downloads
  # (see the PIC toolchain section above).  Without them, the pic* targets
  # skip cleanly and the AVR `make test` is unaffected.  Release/CI use
  # STRICT_TOOLS=1 plus `make pic-test-target-variants`, where any skip fails.

  Mutation tests that require the PIC target toolchain may be allowed to skip
  locally with `MUTATION_ALLOW_SKIP=1` (the default).  Full-toolchain and release
  runs use `MUTATION_ALLOW_SKIP=0` or `STRICT_TOOLS=1`, so a skipped PIC mutant
  is treated as a failed gate.

  Optional ATtiny202 (AVR-XT) dynamic harness (`make attiny202-sim` / -fault /
  -soak; standalone, run locally):
  sudo apt install python3 python3-venv python3-dev g++ curl
  scripts/fetch_attiny_dfp.sh     # Microchip device files (to build the image)
  scripts/fetch_yasimavr.sh       # patched yasimavr venv (to run it)
  # Both are version-pinned, fetched on demand, and NOT committed.  Without them
  # the attiny202-{sim,fault,soak} targets skip cleanly and `make test` is
  # unaffected (see the yasimavr entry under "Simulator" above).

== Version pinning strategy

Exact versions above were the latest Ubuntu 24.04 LTS packages at the time
of initial commit.  Upgrading minor versions of avr-gcc or avr-libc is
generally safe.  If avr-gcc major version changes, re-verify with
`make test` and inspect the generated assembly for the ISR to confirm
the saturating integrator still compiles to the expected atomic read-
modify-write pattern (no multi-byte intermediates).

For CI reproducibility, prefer Ubuntu 24.04 LTS Docker image:

....
    docker run --rm -v $(pwd):/src ubuntu:24.04 bash -c \
      "apt-get update && \
       apt-get install -y make gcc-avr avr-libc simavr libsimavr-dev libelf-dev \
                          gcc clang clang-tidy cppcheck cbmc python3 && \
       cd /src && make test"

The image is minimal: `apt-get update` and `make` itself are required, and
`make test` runs the analyzers (clang-tidy, cppcheck), the MISRA addon (needs
`python3`), and CBMC (`cbmc`) -- omitting any of these makes the corresponding
gate skip silently (or fail under `STRICT_TOOLS=1`). For the authoritative,
tested command set, use `scripts/ci-local.sh` (it mirrors the CI job order).
